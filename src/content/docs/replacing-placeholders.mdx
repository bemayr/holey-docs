    ## Replacing `throw new Error()`

    When a function isn't implemented yet, a common pattern is to throw an error as a placeholder.
    `hole()` makes this intent explicit and trackable.
  </Fragment>
  <Fragment slot="csharp">
    ## Replacing `NotImplementedException`

    When a method isn't implemented yet, a common pattern is to throw an exception as a placeholder.
    `Hole.Todo()` makes this intent explicit and trackable.
  </Fragment>

## Replacing `// TODO` comments

<SlotByFramework>
  <Fragment slot="typescript">
    TODO comments inside function bodies are easy to forget and hard to track.
    `hole()` turns them into a first-class concept.
  </Fragment>
  <Fragment slot="csharp">
    TODO comments inside method bodies are easy to forget and hard to track.
    Holey turns them into a first-class concept.
  </Fragment>
</SlotByFramework>

<SlotByFramework>
  <Fragment slot="typescript">
    <ScrollyCoding steps={[
      {
        description: `<p>A <code>// TODO</code> comment marks missing logic inside a function.</p><p>The function silently returns an empty array — the placeholder is invisible at runtime.</p>`,
        code: `export function listUsers(): Promise<User[]> {
\t// TODO: list users from database
\treturn Promise.resolve([]);
}`,
      },
      {
        description: `<p>Replace the <code>// TODO</code> comment with <code>hole()</code>.</p><p>The gap is now explicit. The <code>return</code> stays — <code>hole()</code> only marks what's missing, not what's there.</p>`,
        code: `export function listUsers(): Promise<User[]> {
\thole("list users from database")
\treturn Promise.resolve([]);
}`,
      },
    ]} />
  </Fragment>
  <Fragment slot="csharp">
    <ScrollyCoding steps={[
      {
        description: `<p>A <code>// TODO</code> comment marks missing logic inside a method.</p><p>The method silently returns an empty list — easy to overlook during review.</p>`,
        code: `public Task<List<User>> ListUsers()
{
\t// TODO: list users from database
\treturn Task.FromResult(new List<User>());
}`,
        diagnostics: [
          { line: 3, severity: 'warning', message: 'HLY101 ⚠ TODO comment should be addressed' },
        ],
      },
      {
        description: `<p>Holey reports the plain <code>// TODO</code> as a warning. You can also add a tag to categorize it as a hole comment.</p><p>The <code>[Database]</code> tag turns it into a tracked hole comment, reported at information severity.</p>`,
        code: `public Task<List<User>> ListUsers()
{
\t// TODO [Database]: list users from database
\treturn Task.FromResult(new List<User>());
}`,
        diagnostics: [
          { line: 3, severity: 'info', message: 'HLY201 ℹ [Database] list users from database' },
        ],
      },
    ]} />
  </Fragment>
</SlotByFramework>

  <Fragment slot="csharp">
    ## Hole types in C#

    Holey for C# recognizes four distinct types of incomplete code, each with its own editor severity:

    | Type | Syntax | Severity |
    |---|---|---|
    | **Todo Comment** | `// TODO: description` | Warning |
    | **Hole Comment** | `// TODO [Tag]: description` | Information |
    | **Missing Implementation** | `throw new NotImplementedException("description")` | Warning |
    | **Side Effect** | `Hole.Todo("description", implementation?)` | Information |
  </Fragment>
